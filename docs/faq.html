FAQ

¶

There are already plenty of template solutions available. Why make another?

A lot of existing template engines force developers into situations where they must combine HTML and processing directives into a single entity. There are a few issues with this approach:

  • The duties of a designer and programmer who deal with markup and JavaScript, respectively, are combined and may not easily be separated during development.
  • The combination of HTML and JavaScript in the one entity will only lead to obtrusive JavaScript, as opposed to unobtrusive JavaScript, which is arguably more desirable.
  • There may be significant overhead in interpreting non-JavaScript directives embedded in HTML.

Additionally, existing solutions may be incorporated in a larger library of code and/or provide extraneous functionality which is not directly related to templating. A good example is the use of AJAX built in to the code to retrieve template data. That is not to say those features are not useful to have, but rather could be provided by another, potentially custom, API.

Penguin aims to be able to provide a clear-cut separation between the markup and JavaScript used in templating, while being minimal but powerful enough for developers to be as expressive as they please.

¶

eval is evil. Why is it used in Penguin?

If a logical component is given for a precompiled template, it must be converted from a string as defined in HTML, to a native JavaScript function. This is done with eval(). However, eval()'s use is contentious in the JavaScript community as it permits the execution of an arbitrary string of JavaScript. However, as long as the string has been generated by a trusted source (a server, for instance), it is no less secure than any other JavaScript, such as those already included with the web page. You should be more concerned with man-in-the- middle attacks instead. A more legitimate cause for concern with eval() is how inefficient it is because of the need to invoke the JavaScript interpreter. In Penguin however, eval() is only used to define a function. It is not used for any other purpose or to execute any other code. Additionally, if a template needs to be copied (which may occur when DataBinding a collection), a copy of the function object is made; the string used for the function is not passed back to the interpreter. Regardless, any overhead incurred by eval() in Penguin is considered negligible and its use justified given the functionality it provides.

What about browser compatibility?

Penguin has been confirmed to work in the following browsers:

  • Google Chrome 31
  • Firefox 25
  • Opera 12
  • Internet Explorer 11
  • Safari 5 (mobile)
